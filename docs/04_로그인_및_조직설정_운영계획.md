# BoanClaw 로그인 및 조직설정 운영 계획

버전: v0.1.0  
작성일: 2026-03-30  
대상: BoanClaw Desktop App, Admin Console, 조직정책서버

---

## 1. 목표

BoanClaw의 로그인과 조직설정 운영은 `Claude Desktop`와 유사하게 가져간다.

- 사용자는 앱에서 간단히 로그인한다.
- 조직 정책, 허용 모델, SSO 구성, seat 제한, 버전 통제는 중앙의 `조직정책서버`가 관리한다.
- 데스크톱 앱은 정책 서버에 체크인해서 기동 허용 여부와 현재 조직 정책을 받는다.
- 로그인 자체와 조직 권한은 분리한다.

즉, `인증은 SSO/OAuth`, `운영 제어는 조직정책서버`로 나눈다.

---

## 2. 목표 UX

### 사용자 UX

1. BoanClaw Desktop 실행
2. 회사 이메일 입력
3. 조직 정책에 따라 아래 중 하나로 로그인
   - Google OAuth
   - 이메일 OTP
   - 향후 OIDC / SAML SSO
4. 로그인 성공 후 `org_id`, `role`, `workspace_url`, `allowed_models`를 수신
5. 앱 기동 시 정책 서버 `checkin` 수행
6. 정책 허용 시에만 에이전트 기능 활성화

### 관리자 UX

1. Admin Console 로그인
2. 조직 선택
3. 조직설정 탭에서 아래 항목 관리
   - 허용 로그인 방식
   - 관리자 이메일
   - GCP Org ID
   - seat 제한
   - 허용 모델
   - 기능 플래그
   - 최소 버전 / 차단 버전
4. 저장 시 정책 서버에 새 버전 발행
5. Desktop App은 다음 체크인 시 새 정책 반영

## 4. 조직정책서버를 진실의 원천으로 두는 이유

로그인 시스템만으로는 아래를 통제할 수 없다.

- 특정 버전 강제 업데이트
- 차단된 모델 사용 금지
- seat 제한
- 조직별 허용 SSO 방식
- 조직별 GCP 연동 정보
- 네트워크 allowlist
- 원격 기능 플래그

따라서 로그인 후에도 앱은 반드시 조직정책서버에 `checkin`해야 한다.

---

## 5. 정책 서버 API 계획

### 현재 구현 기준

- `GET /org/{org_id}/policy.json`
- `POST /org/{org_id}/policy`
- `GET /org/{org_id}/network-policy.json`
- `GET /org/{org_id}/v1/policy`
- `POST|PUT /org/{org_id}/v1/policy`
- `GET /org/{org_id}/v1/org-settings`
- `PATCH|PUT /org/{org_id}/v1/org-settings`
- `POST /org/{org_id}/v1/checkin`

### checkin 응답 의미

- `allowed`
  - 현재 버전 사용 허용
- `update_required`
  - 최소 버전보다 낮음
- `blocked`
  - 조직 정책상 명시 차단

정책 서버는 응답에 정책 본문과 서명을 같이 넣고, 앱은 내장 공개키로 검증해야 한다.

## 6. Claude Desktop 스타일 운영 원칙

BoanClaw는 아래 원칙을 따른다.

- 로그인은 사용자가 쉽게 해야 한다.
- 조직 정책은 중앙에서 바꿔야 한다.
- 앱은 로컬 설정만 믿지 않는다.
- 정책 서버가 응답하지 않으면 보안 민감 기능은 fail-closed 한다.
- 조직 변경, 로그인 방식 변경, seat 제한은 콘솔에서 바꾸고 앱은 체크인으로 반영한다.

즉 데스크톱 앱은 `세션 소비자`, 정책 서버는 `조직 운영 제어면`이다.

## 9. 로컬 Admin Console 배포 기준

로컬 개발에서 브라우저로 여는 Admin Console 주소는 아래로 고정한다.

- `http://localhost:19080/login`

이 주소는 `docker-compose.dev.yml`의 `boan-admin-console` 서비스가 제공한다.

- `boan-admin-console`: `19080 -> 80`
- `boan-proxy`: `18080`, `18081`
- `boan-policy-server`: `8081`

즉, 로그인 화면 문자열 수정이나 OTP 안내 문구 수정은 `코드만 바꾸면 끝이 아니고`, 반드시 로컬 컨테이너 재빌드/재배포까지 해야 브라우저에 반영된다.

### 로컬 재배포 명령

```bash
cd /home/dowoo/desktop/dowoo/SDS-RED/SDSclawBLUE/BoanClaw
docker compose -f docker-compose.dev.yml up -d --build boan-proxy boan-admin-console
```

화면이 그대로면 프런트를 강제 재생성한다.

```bash
cd /home/dowoo/desktop/dowoo/SDS-RED/SDSclawBLUE/BoanClaw
docker compose -f docker-compose.dev.yml up -d --force-recreate boan-admin-console
```

컨테이너 상태 확인:

```bash
docker ps --format '{{.Names}}\t{{.Status}}' | rg 'boanclaw-boan-(admin-console|proxy|policy-server)'
```

주의:

- `http://localhost:19080/login`은 Cloud Run이 아니라 로컬 Docker Compose 화면이다.
- Cloud Run에 배포해도 로컬 `19080`은 자동으로 바뀌지 않는다.
- 로컬 화면 기준 수정 확인은 반드시 로컬 컨테이너 재빌드 후 확인한다.

## 10. 배포 습관

로그인/OTP/UI 문구/프록시 응답을 수정했으면 아래 순서를 기본 절차로 삼는다.

1. 소스 수정
2. 테스트 실행
3. 로컬 Docker Compose 재빌드
4. 브라우저에서 `http://localhost:19080/login` 확인
5. 원격 Cloud Run 재배포가 필요한 변경이면 별도로 배포

즉, `BoanClaw 로그인 화면 수정 = 코드 수정 + 빌드 + 로컬 재배포 + 화면 확인`까지 완료해야 끝난 것으로 본다.

## 11. 현재 운영 규칙

현재 BoanClaw 로그인/조직 편입 규칙은 아래로 고정한다.

- 고정 소유자 이메일: `genaisec.ssc@samsung.com`
- 소유자는 1명으로 고정한다.
- 소유자 권한은 배포 설정값(`BOAN_OWNER_EMAIL`)으로만 정한다.
- 다른 사용자는 `owner`가 될 수 없다.
- 다른 사용자가 Google SSO 또는 이메일 OTP로 처음 로그인하면 즉시 조직에 편입되지 않는다.
- 첫 로그인은 `사용 요청`으로 저장되고 상태는 `pending`이다.
- 소유자가 사용자 관리 화면에서 승인하면 `approved`로 바뀌고 이후 로그인 가능하다.
- 소유자는 언제든 사용자를 삭제할 수 있다.
- 조직설정 화면은 소유자만 접근할 수 있다.

즉 현재 권한 모델은 `소유자(owner)` / `사용자(user)` 2개뿐이다.
