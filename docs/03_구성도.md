# BoanClaw 구성도

> 영역(S1–S4) · 게이트(G1–G3) · 정보 흐름 엣지 · 기본 마운트 경로 · 모든 컨테이너.
> 이 문서가 BoanClaw 의 single source of truth. 코드와 docker-compose 에서 직접 추출했음.

## 2. 기본 마운트 경로 (코드에서 그대로 추출)

### 2-1. 호스트 ↔ Sandbox

| 호스트 (S3) | Sandbox (S2) | 종류 | 출처 |
|---|---|---|---|
| `~/Desktop/boanclaw/` | `/home/boan/Desktop/boanclaw/` | bind mount (read-write) | `BOAN_HOST_MOUNT_ROOT` → `BOAN_MOUNT_ROOT` |
| `${BOAN_GCP_SERVICE_ACCOUNT_KEY_PATH}` | `/var/run/gcp/service-account.json` | bind mount (ro) | `GOOGLE_APPLICATION_CREDENTIALS` |
| (named volume) | `/data/users` | named volume `boan-user-data` | 사용자 세션 데이터 |
| (named volume) | `/data/registry` | named volume `boan-registry-data` | LLM endpoint registry |
| (named volume) | `/data/rdp-transfer` | named volume `boan-rdp-transfer` | **S2 ↔ S1 RDP 가상드라이브 staging — guacd 와 공유** |

### 2-2. S4 격리 (sandbox 절대 마운트 안 됨)

| Volume | 마운트 위치 | 어떤 컨테이너만? | 용도 |
|---|---|---|---|
| `boan-cred-keys` | `/etc/boan-cred` | `boan-credential-filter` 만 | AES 256 키 (`aes.key`) |
| `boan-cred-data` | `/data/credentials` | `boan-credential-filter` 만 | 암호화된 credential JSON |
| `boan-policy-keys` | `/etc/boan-policy` | `boan-policy-server` 만 | 정책 서명 키 (Ed25519) |
| `boan-policy-data` | `/data/policies` | `boan-policy-server` 만 | 정책 / 헌법 / training log |

> sandbox 의 `docker-compose.dev.yml` volumes 절에 **`boan-cred-*`, `boan-policy-*` 가 절대 들어가지 않음** — Anthropic 권고 "credential 을 sandbox 에서 구조적으로 분리" 를 코드 레벨로 강제.

### 2-3. RDP virtual drive 흐름

```
Windows VM (S1)                                            sandbox (S2)
  BoanClaw drive  ──── guacd ────────────────  /data/rdp-transfer/<email>/
                   (RDP virtual channel,                    │
                   같은 named volume)                       │
                                                             │
                                                       boan-proxy 가 watch
                                                             │
                                                             ▼
                                            /home/boan/Desktop/boanclaw/
                                                    (S2 작업 공간)
                                                             │
                                                       (bind mount)
                                                             │
                                                             ▼
                                                  ~/Desktop/boanclaw/
                                                       (호스트, S3)
```

guacd 는 `BOAN_UID:BOAN_GID` 와 동일하게 실행 (compose `user: "${BOAN_UID:-1000}:${BOAN_GID:-1000}"`) — sandbox 의 boan-proxy 가 RDP staging dir 의 owner 자격으로 양방향 read/write 가능. `HOME=/tmp` 로 FreeRDP TLS 인증서 캐시도 정상 동작.

## 3. 정보 흐름 — Edge Graph

각 화살표가 **정보 흐름 1 단위**. 게이트는 노드 위에 라벨로 표시.

### 3-1. 키보드 · 클립보드 · 채팅 입력 (사용자 → S1)

```
                               ┌──────────────────────────────────────┐
   사용자 브라우저              │            boan-proxy (S2 내부)       │
   admin-console (S3)          │                                      │
        │                      │     ┌──────────────────────────────┐ │
        │  HTTP /api/input-    │     │  Input Gate 평가 파이프라인  │ │
        │  gate/evaluate       │     │                              │ │
        │  body: {text, mode,  │     │   ┌── G1 ──┐  정규식 검사    │ │
        │         user}        │     │   │        │  (전원 적용)    │ │
        │                      │     │   └────┬───┘                 │ │
        ▼                      │        match │ no-match            │ │
   ┌────────────┐  POST        │        ↓     ▼                     │ │
   │ boan-      │ ───────────► │   credential_required               │ │
   │ admin-     │              │        ↓                            │ │
   │ console    │              │   user.access_level                 │ │
   │            │              │   ┌── allow ──┐ ┌── ask/deny ──┐   │ │
   └────────────┘              │   │           │ │              │   │ │
                               │   ▼           │ ▼              │   │ │
                               │ DLP only      │ ┌──── G2 ────┐ │   │ │
                               │ 통과          │ │ 헌법+LLM   │ │   │ │
                               │               │ │ allow→pass │ │   │ │
                               │               │ │ block→stop │ │   │ │
                               │               │ │ ask→G3     │ │   │ │
                               │               │ └─────┬──────┘ │   │ │
                               │               │       ▼        │   │ │
                               │               │ ┌──── G3 ────┐ │   │ │
                               │               │ │ Wiki LLM   │ │   │ │
                               │               │ │ + train log│ │   │ │
                               │               │ │ allow→pass │ │   │ │
                               │               │ │ block→stop │ │   │ │
                               │               │ │ ask→ HITL  │ │   │ │
                               │               │ └─────┬──────┘ │   │ │
                               │               │       ▼        │   │ │
                               │               │  Approvals 큐  │   │ │
                               │               │   (소유자 승인)│   │ │
                               │               └────────────────┘   │ │
                               │                       │            │ │
                               │                       ▼ approved   │ │
                               │                   sandbox 로 forward│ │
                               │                                    │ │
                               │   DLP / credential 치환             │ │
                               │   {{CREDENTIAL:name}} → 실제 값    │ │
                               │                                    │ │
                               │   audit-agent 로 trace 1 줄 emit   │ │
                               └─────────────────┬──────────────────┘ │
                                                 │                    │
                                                 ▼                    │
                                       sandbox / S1 (Windows)         │
```

`Approvals 큐` 는 admin-console 의 *Approvals > User Actions* 페이지에서 소유자가 승인/거부. 거부 시 trace 에 `decision=block, gate=G3, source=human` 으로 남음.

### 3-2. Credential Gateway (S4 격리)

```
사용자 프롬프트
"curl -H 'Authorization: Bearer {{CREDENTIAL:my-api-key}}' ..."
        │
        ▼
boan-proxy (S2)
   · 입력에서 {{CREDENTIAL:name}} 패턴 추출
   · 동시에 raw 키 패턴(sk-*, ghp_*, AKIA*, …) 도 정규식 스캔
        │
        ├── raw 키 발견 → G1 차단 (credential_required)
        │
        └── placeholder 만 있음 → 다음 단계
                │
                ▼
          ┌─────────────────────────────────┐
          │ HTTP GET                        │
          │ http://boan-credential-filter:  │
          │      8082/credential/{org}/{name}│
          └─────────────┬───────────────────┘
                        │
                        ▼
          ┌─────────────────────────────────┐
          │  boan-credential-filter (S4)    │
          │  ─────────────────────────────  │
          │  /etc/boan-cred/aes.key 읽음    │
          │  /data/credentials/{name}.enc   │
          │  AES 복호화                      │
          │  → plaintext 반환 (HTTP body)   │
          └─────────────┬───────────────────┘
                        │
                        ▼
                boan-proxy 가 메모리에서만 치환
                · 디스크에 안 씀
                · 로그에는 placeholder 만 남김
                · audit-agent trace: type=credential, gate=credential
                        │
                        ▼
                  외부 LLM API 호출 (S1)
                  (Network Gate 도 통과해야 함)
```

**S4 격리 핵심**: sandbox 는 `boan-cred-keys`/`boan-cred-data` 볼륨을 절대 마운트하지 않음. raw key 는 오직 `boan-credential-filter` 컨테이너 메모리에서만 등장. sandbox 침투당해도 AES 키가 없어서 vault 를 복호화 못 함.

### 3-3. Network Gateway (egress 화이트리스트)

```
sandbox 내부 임의 프로세스 (예: openclaw, boan-agent)
        │
        │ HTTP_PROXY=http://localhost:18080
        ▼
   boan-proxy (S2 내부, MITM)
        │
        ▼
   ┌────────────────────────────────────┐
   │  Network Gate                      │
   │  ─────────────────────────────────│
   │  policy = boan-policy-server 에서  │
   │           주기적으로 fetch          │
   │  policy.allowlist = [              │
   │     {host:"api.anthropic.com",    │
   │      port:443, methods:["POST"]}, │
   │     {host:"api.openai.com", …},   │
   │     ...                            │
   │  ]                                 │
   │                                    │
   │  match? ── yes ──→ forward         │
   │       └── no  ──→ 403 + audit      │
   │                                    │
   │  CONNECT 터널 → 항상 차단          │
   └────────────────────────────────────┘
        │
        │ (allowed only)
        ▼
   외부 API (S1)
```

`whitelist` 가 비어 있거나 정책 서버가 안 닿으면 **모든 호출 block** (fail-closed). 호스트가 허용 목록에 있어도 method/port 가 다르면 차단.

### 3-4. 파일 전송 (File Manager)

```
admin-console File Manager
       │
       │ user 가 drag & drop
       │
   ┌───┴────┐
   │        │
   ▼        ▼
S2 (좌)   S1 (우)
   │        │
   │        │
   │        │  ┌───────────────┐
   │        └─►│ S1 → S2       │  높→낮: 검사 면제 (단, 파일만, 폴더 X)
   │           │ 무조건 통과    │
   │           └───────┬───────┘
   │                   │
   │                   ▼
   │        sandbox /home/boan/Desktop/boanclaw 로 복사
   │        ↓
   │        호스트 ~/Desktop/boanclaw 에서도 즉시 보임 (bind mount)
   │
   │  ┌──────────────────────────────────┐
   └─►│ S2 → S1                          │  낮→높: Input Gate 평가
      │ ────────────────────────────────│
      │ 1. 파일 헤드 1 MB 추출            │
      │ 2. G1 정규식 (credential 패턴)   │
      │ 3. (ask/deny 사용자) G2 → G3     │
      │ 4. 통과 시 /data/rdp-transfer/   │
      │    <email>/ 로 복사               │
      │    → guacd 가 RDP virtual drive  │
      │      에 노출 → Windows 측 BoanClaw│
      │      드라이브로 보임              │
      └──────────────────────────────────┘
```

S2→S1 은 **파일만** (폴더 차단). S1→S2 는 같은 named volume 의 inode 를 sandbox 가 직접 read 해서 sandbox FS 로 복사.

### 3-5. Computer-Use (Vision → Action 루프)

```
사용자 (admin-console "BoanClaw" / "Personal Computer" 메뉴)
   │
   │ "이 화면에서 X 를 클릭해줘"
   ▼
boan-computer-use (Playwright 기반)
   │
   ├── screenshot()  ── RDP 화면 캡처
   │
   ▼
Vision LLM (LLM Registry 의 Security LMM 바인딩)
   │
   ├── 스크린샷 → 어디 클릭/타이핑할지 결정
   │
   ▼
큐로 publish:  type / key / click / scroll
   │
   ▼
admin-console front-end poll → user-facing 큐 → boan-proxy /api/computer-use/execute
   │
   ▼
   ┌──────────────────────────────────────┐
   │ Input Gate 검사 (type, key 액션만)    │
   │  · text → G1/G2/G3 (위 3-1 과 동일)  │
   │  · 안전 키 (Tab, Enter 등) → 통과    │
   └─────────────┬────────────────────────┘
                 │
                 ▼
            Guacamole RDP 채널 → Windows VM 에 입력 주입
```

**모든 type/key 액션이 Input Gate 를 거친다** — 에이전트가 화면을 통해 우회로 비밀을 송출하는 시나리오 차단.

### 3-6. 인증 / SSO

```
사용자 → /login
   │
   ├── Email OTP
   │     · 도메인 검증 (org_settings.allowed_domains)
   │     · OTP 발송 → 입력 → 세션 생성
   │
   └── Google OAuth
         · hd= 파라미터로 도메인 강제
         · 콜백에서 도메인 재검증
         · 세션 생성

신규 사용자: status=pending
   ↓
소유자가 Users 패널에서 승인 → status=approved + GCP workstation 할당
   ↓
사용자별 access_level: allow / ask / deny → G2/G3 적용 여부 결정
```

## 4. 컨테이너 일람 (`docker-compose.dev.yml` 기준)

| # | 컨테이너 | 영역 | 포트 | 핵심 책임 |
|---|---|---|---|---|
| 1 | `boan-guac-db` | S3 | (internal) | Guacamole 메타데이터 (Postgres) |
| 2 | `boan-guacd` | S3 | (internal) | RDP / VNC 프록시. virtual drive bridge |
| 3 | `boan-guacamole` | S3 | 8080 | RDP web 콘솔 (admin-console iframe) |
| 4 | `boan-policy-server` | S4 | 8081 | 정책 store, G2/G3 evaluate endpoint, 헌법 개정 |
| 5 | `boan-credential-filter` | S4 | 8082 | AES vault. sandbox 에서 마운트 금지 |
| 6 | `boan-audit-agent` | S4 | 8084 | OTEL trace 수집, ring buffer (500개) |
| 7 | `boan-whitelist-proxy` | S3 | 8085 | (legacy) 외부 egress 화이트리스트 프록시 |
| 8 | `boan-llm-registry` | S3 | 8086 | LLM endpoint DB, security binding 관리 |
| 9 | `boan-asset-constitution` | S3 | 8087 | 자산/헌법 메타 |
| 10 | `boan-proxy` | (legacy host profile) | 18080/18081 | MITM proxy. dev 에서는 sandbox 내장본 사용 |
| 11 | `boan-computer-use` | S2 보조 | 8090 | Playwright 화면 자동화 |
| 12 | `boan-admin-console` | S3 | 19080 | React UI |
| 13 | `boan-sandbox` | S2 | 18789 | OpenClaw + boan-agent + 내장 boan-proxy + onecli |

`boan-proxy` 는 두 군데에서 등장: ① compose 의 `boan-proxy` 서비스 (`profiles: [legacy-host]` — 일반 dev 에서 비활성), ② `boan-sandbox` 컨테이너 안에 빌드 시점 임베드되어 entrypoint 가 `gosu boan boan-proxy` 로 띄움. **메모리 규칙**: 두 곳을 항상 함께 빌드 — `scripts/rebuild.sh` 가 자동 처리.

## 5. OpenClaw Supply-Chain 검증

```
빌드 시점 (Dockerfile)                        런타임 (entrypoint.sh)
─────────────────────────                     ─────────────────────────
ARG OPENCLAW_VERSION=2026.4.9                 expected = cat /opt/boanclaw-meta/openclaw.version
       │                                      expected_hash = cat /opt/boanclaw-meta/openclaw.mjs.sha256
       ▼
npm install -g openclaw@${OPENCLAW_VERSION}        actual = node -e "..."  ← 실제 package.json
       │                                          actual_hash = sha256sum openclaw.mjs
       ▼
node -e "package.json.version" 대조                       │
       │                                                  ▼
       ├── mismatch → image build FAIL                ┌───┴──────────────┐
       │                                              │ expected_version │
       ▼                                              │ != actual?       │ ── yes ──► fail-closed exit
sha256sum openclaw.mjs                                │                  │
       │                                              │ expected_hash    │
       ▼                                              │ != actual?       │ ── yes ──► fail-closed exit
/opt/boanclaw-meta/openclaw.version                   │                  │
/opt/boanclaw-meta/openclaw.mjs.sha256                │ allowlist 있고   │
                                                      │ 포함 안 됨?       │ ── yes ──► fail-closed exit
                                                      └────┬─────────────┘
                                                           │ all OK
                                                           ▼

```

추가 레이어: `BOAN_OPENCLAW_ALLOWED_VERSIONS` 환경변수 (콤마구분) — 정책 서버나 운영자가 동적으로 좁힐 수 있음. 예: 새 컨테이너만 핀 버전을 허용하고, 다른 사이트는 이전 버전도 허용.

## 7. Fail-Closed 매트릭스

| 컴포넌트 실패 | 동작 |
|---|---|
| G1 (정규식) | 항상 동작 — network 의존성 없음 |
| G2 LLM 미응답 | **block** (heuristic fallback 아님) |
| G3 LLM 미응답 | **block** |
| boan-policy-server 미응답 | 정책 fetch 실패 → 캐시 만료 후 **block** |
| boan-credential-filter 미응답 | credential resolve 실패 → 호출 자체 **block** |
| Network Gate 정책 비어있음 | 모든 outbound **block** |
| OpenClaw hash mismatch | sandbox 컨테이너 즉시 종료 (healthcheck 실패) |

## 8. 영역간 화살표 요약

```
S1 ──→ S2 :  파일 (검사 면제), RDP 입력 (Input Gate 적용 — 거꾸로 흐르는 정보)
S2 ──→ S1 :  파일/채팅/Computer-Use (Input Gate G1/G2/G3 + Credential Gate + Network Gate)
S2 ──→ S4 :  HTTP API only (credential resolve, policy fetch, audit emit)
S3 ──→ S4 :  HTTP API only (admin 명령, 정책 변경, training-log feedback)
S4 ──→ * :  outbound 자체 없음 (수동적 응답만)
```

영역 화살표는 단방향 게이트가 아니라 **항상 게이트를 통과한 후** 흐름이 일어남. fail-closed.
