# BoanClaw 통합 테스트 명세

> 호스트에서 실행. 각 UI 기능을 실제 API 호출로 검증하고 항상 원상복구.
> Computer-Use 관련 기능은 **이 명세의 대상이 아님** (별도 수동 QA).

---

## 0. 운영 모델

```
호스트 (test runner)
   │
   │  curl -b cookie.jar  http://localhost:18081/api/test/...
   ▼
boan-proxy admin (sandbox 내부, :18081)
   │
   ├── 일반 API 와 같은 mux
   └── /api/test/* 는 cfg.TestMode == true 일 때만 등록됨
        │
        ├─ /api/test/status         (test 모드 활성 여부 + version)
        ├─ /api/test/session        (임의 email/role/access_level 로 세션 발급)
        ├─ /api/test/sandbox-exec   (sandbox 안에서 sh -c 실행 — network gate 검증)
        ├─ /api/test/policy-raw     (policy store 직접 dump)
        └─ /api/test/cleanup-user   (테스트 사용자 강제 삭제)
```

`TEST=true` 환경 변수는 이미 `docker-compose.dev.yml` 의 `boan-sandbox.environment` 에 박혀있고, prod compose 에서는 빼면 됨. test-mode endpoint 들은 그 변수가 false 일 때 등록 자체가 안 되므로 prod 에서 fingerprintable 해도 상관없음.

> ⚠️ 보안 경고: `/api/test/sandbox-exec` 는 임의 shell 실행이라 매우 위험한 권한.
> 반드시 `TEST=true` 로 부트한 dev 컨테이너에서만 활성화. prod 배포 시 compose 에서 `TEST` 환경변수 제거 필수.

## 1. 테스트 surface — 기능별 매트릭스

| # | 영역 | 기능 | API 엔드포인트 | 검증 방법 | Cleanup |
|---|---|---|---|---|---|
| 1 | Auth | test 세션 발급 | `POST /api/test/session` | `GET /api/auth/me` 가 200 + 동일 email 반환 | 세션 쿠키 폐기 |
| 2 | Users | CRUD | `POST/GET/DELETE /api/admin/users` | list 에 등장 → 권한 변경 → list 에서 제거 | DELETE |
| 3 | Users | access_level 토글 | `PATCH /api/admin/users/{email}` (`access_level`) | get 으로 반영 확인 | 원상복구 |
| 4 | LLM Registry | 등록/조회/삭제 | `POST/GET/DELETE /api/registry/...` | 등록 후 GET 에 보임 | DELETE |
| 5 | Network Policy | 호스트 추가 → curl 로 통과 확인 | `POST /api/policy/...` + `POST /api/test/sandbox-exec {cmd: "curl -s -o /dev/null -w '%{http_code}' https://api.example.com/"}` | exit_code == 0 + 200 응답 | 호스트 제거 → 다시 sandbox-exec → 차단 확인 |
| 6 | Network Policy | 비허용 호스트 차단 확인 | `POST /api/test/sandbox-exec {cmd: "curl -sf https://evil.example.com/"}` | exit_code != 0 (403 또는 connection refused) | (없음 — read-only) |
| 7 | Mount Policy | 설정 변경 | `POST /api/mount/config` | GET 으로 반영 | 원래 값 복구 |
| 8 | Guardrail Policy | 헌법 변경 | `POST /api/policy/{org}/guardrail` | GET 으로 반영 | 원래 헌법 복구 |
| 9 | Credentials | 등록/resolve/삭제 | `POST /api/credential/v1/store` + `GET /api/credential/{org}/{name}` | resolve 시 같은 값 반환 (값은 cloud Secret Manager 에서 옴) | DELETE |
| 9a | Credentials revoke (P4) | `POST /api/credential/v1/revoke` → 동일 role resolve | 등록 → revoke → resolve 404 + `credential_revoke` 감사 이벤트 | (revoke 후 자동) |
| 10 | Credential Vault 격리 (로컬 쪽) | sandbox 에서 키 파일 접근 불가 | `POST /api/test/sandbox-exec {cmd: "ls /etc/boan-cred/ 2>&1"}` | "No such file or directory" | (없음) |
| 10a | Credential Vault 격리 (로컬 filter) | credential-filter 컨테이너 디스크 상태 | `POST /api/test/sandbox-exec {cmd: "docker exec boan-credential-filter cat /data/credentials/credentials.json 2>&1 \| wc -c"}` | 등록 전후 파일 크기 변동 없음 (신규는 cloud 로만) | (없음) |
| 10b | Placeholder passthrough | LLM 호출 envelope 검증 | boan-proxy 가 org-llm-proxy 에 보내는 body 에 `{{CREDENTIAL:*}}` 그대로 유지 | 치환은 Cloud Run 에서만 | (없음) |
| 11 | Approvals | 큐 조회 + 승인 | `GET /api/approvals` + `POST /api/approvals/{id}` | 큐에 등장 → 승인 후 사라짐 | (자동) |
| 12 | Audit Log | trace 조회 | `GET /api/observability/traces` | 위 테스트들이 실행한 동작이 trace 에 보임 | (read-only) |
| 13 | SSO | 도메인 추가/삭제 | `POST /api/admin/org-settings` | `GET /api/auth/config` 의 `allowed_email_domains` | 원래 값 복구 |
| 14 | File Manager | S2 list | `GET /api/files/list?side=s2` | 파일 등장 | (없음) |
| 15 | File Manager | S2→S1 안전 파일 전송 | `POST /api/files/transfer` | 200 + S1 list 에 등장 | S1 의 파일 삭제 |
| 16 | File Manager | S2→S1 위험 파일 차단 | 동일, 본문에 `sk-ant-xxxxxxxxxxxxxxxx` | error: "credential" | (전송 안 됨) |
| 17 | Input Gate G1 | 정규식 차단 | `POST /api/input-gate/evaluate` | `decision: credential_required` | (read-only) |
| 18 | Input Gate G2 | LLM 차단 (mock) | 동일 | `decision: block` (정책 LLM 미설정 → fail-closed) | (read-only) |
| 19 | OpenClaw 무결성 | meta 파일 존재 | `POST /api/test/sandbox-exec {cmd: "test -f /opt/boanclaw-meta/openclaw.version && echo OK"}` | OK | (없음) |
| 20 | OpenClaw 무결성 | sha256 일치 | `POST /api/test/sandbox-exec {cmd: "..."}` | 일치 | (없음) |

> File Manager S1↔S2 의 양방향 권한 검증 (uid alignment) 은 16번 케이스로 자동 커버됨 — 전송이 권한 문제 없이 성공해야 함.

## 2. 테스트 러너 사용법

```bash
# 모든 테스트 (cleanup 보장)
./scripts/test-integration.sh

# 특정 테스트만
./scripts/test-integration.sh test_network_policy_block

# 첫 실패에서 stop
FAIL_FAST=1 ./scripts/test-integration.sh
```

각 테스트 함수는:
1. `setup` 단계에서 필요한 최소 상태 만들기
2. `assert` 단계에서 동작 검증
3. `cleanup` 단계는 trap 또는 명시 호출로 항상 실행 (실패해도 부수효과 안 남기기)

## 3. test-mode endpoint 명세

### `GET /api/test/status`
- 응답: `{"test_mode": true, "version": "...", "openclaw_version": "..."}`
- 가드: 없음 — test 모드에서만 등록되므로 존재 자체가 곧 test 모드

### `POST /api/test/session`
- body: `{"email": "test@example.com", "role": "owner"|"user", "access_level": "allow"|"ask"|"deny"}`
- 동작: 사용자가 없으면 Upsert → SetAccessLevel → 세션 토큰 발급 → 쿠키 set + body 에도 토큰 반환
- 응답: `{"ok": true, "token": "...", "email": "...", "role": "...", "access_level": "..."}`

### `POST /api/test/sandbox-exec`
- body: `{"cmd": "curl -s ..."}` (또는 `{"argv": ["sh","-c","..."]}`)
- 동작: `sh -c <cmd>` 실행, stdout/stderr/exit_code 캡처. 10초 timeout. stdout/stderr 각 16KB 제한.
- 응답: `{"ok": true, "stdout": "...", "stderr": "...", "exit_code": 0, "duration_ms": 123}`
- 사용 예 — network gate 통과 검증:
  ```
  curl -X POST http://localhost:18081/api/test/sandbox-exec \
    -H 'Content-Type: application/json' \
    -d '{"cmd":"curl -s -o /dev/null -w \"%{http_code}\" https://api.anthropic.com/v1/messages"}'
  ```
  Network policy 에 `api.anthropic.com:443` 이 없으면 403 또는 connection refused.

### `GET /api/test/policy-raw?org=...&kind=network|guardrail|mount`
- 응답: `{...정책 JSON 그대로...}`
- 정책 fetch / 캐시 / 서명 검증을 우회한 raw store dump

### `POST /api/test/cleanup-user`
- body: `{"email": "..."}`
- 동작: userstore 에서 강제 Delete (Authentication 우회)
- 응답: `{"ok": true}`

## 4. 테스트 검사 항목 우선순위

1. **P0** — 기본 sanity: status, session, sandbox-exec 가 동작
2. **P0** — Network gate: 허용/차단/롤백 (사용자 핵심 우려)
3. **P0** — Credential vault 격리 (Anthropic 원칙)
4. **P0** — Input gate G1 정규식
5. **P1** — Users CRUD, access_level
6. **P1** — File Manager S2↔S1 (최근 수정한 perms 검증 포함)
7. **P1** — Guardrail policy, Mount policy CRUD
8. **P2** — LLM Registry, Approvals, Audit
9. **P2** — SSO 도메인
10. **P3** — OpenClaw 무결성 (이미 entrypoint 가 검사하므로 외부 검증은 보조)
11. **P0** — Cloud LLM egress (P1): boan-proxy 가 외부 LLM 으로 직접 나가지 않음
12. **P0** — Cloud credential vault (P2): 로컬 평문 저장 없음, Secret Manager 가 truth source
13. **P1** — Device JWT (P3): Cloud Run 이 JWT 없이 호출 거부
14. **P1** — Revoke + rate limit (P4): 감사 이벤트 emit 확인

## 5. 알려진 제약

- Test runner 는 sandbox 컨테이너가 떠 있어야 동작 (`docker compose ps` 로 확인)
- guardrail G2/G3 LLM 미설정 환경에서는 G2 가 자동으로 fail-closed → block 으로 떨어짐 (의도된 동작)
- File Manager S2→S1 검사는 비-trivial — 파일 내용 헤드 1MB 를 G1 정규식으로 스캔
- Computer-Use 는 명시적으로 제외 (Vision LLM + Playwright 의존성)
