# BoanClaw 보안 철학

## 1. S5 구조 — 정보의 영역 분리

```
S5  Cloud Vault + Egress ── GCP Cloud Run: credential vault (Secret Manager), LLM egress broker
S4  Cloud Policy Zone ────── GCP Cloud Run: 조직 정책, 감사 로그, 서명 키, guardrail LLM
S3  Host Zone ────────────── 사용자 PC, 관리 콘솔, 프록시 (thin forwarder)
S2  Sandbox Zone ─────────── Docker 격리 환경, OpenClaw 에이전트, 코드 실행
S1  External Zone ────────── GCP 작업 PC, 외부 LLM API, 인터넷
```

## 2. 정보 흐름 원칙

```
높은 → 낮은 (S5→S1): 검사 — G1/G2/G3 가드레일 + DLP + Credential Gate + egress allowlist
낮은 → 높은 (S1→S5): 통과 — 모니터링만
```

## 3. 내부/외부 경계 — Credential 격리 (P2 이후)

```
┌──────────────────────────────────────────────┐
│  S5: Cloud Credential Vault (GCP Cloud Run)  │  ← 실제 API 키 평문은 여기만 존재
│       boan-org-credential-gate               │     Secret Manager secret: boan-cred-{org}-{role}
│       + Secret Manager                        │     로컬 호스트는 접근 불가 (bearer / JWT 필요)
│                                              │
│  S5: Cloud LLM Egress Broker                 │  ← 모든 외부 LLM 호출의 유일한 네트워크 egress
│       boan-org-llm-proxy                     │     {{CREDENTIAL:role}} 을 LLM 호출 직전에만 치환
│                                              │     upstream response 에서 credential echo scrub
│                                              │
│  S4: Cloud Policy (GCP Cloud Run)            │  ← 정책, 가드레일, wiki, 감사 로그
│       boan-policy-server                     │     조직별 bearer token 으로 gate
└──────────┬───────────────────────────────────┘
           │ HTTPS bearer (P3 에서 device-signed JWT 로 교체)
           ▼
┌──────────────────────────────────────────────┐
│  S3: 관리 콘솔, 프록시 (평문 credential 없음)  │  ← credential-filter 는 thin forwarder
│       boan-proxy — placeholder 만 다룸        │     로컬 AES 키 / credentials.json 은 legacy fallback
│       boan-credential-filter — gate mode     │     신규 등록은 로컬 디스크 건너뜀
│  S2: Sandbox (평문 credential 없음)           │  ← HTTP API 호출 (placeholder) 만 가능
└──────────┬───────────────────────────────────┘
           │ Network Gate (fail-closed) — 외부 egress 는 반드시 S5 경유
           ▼
│  S1: 외부 LLM, GCP 작업 PC                    │
```

**Anthropic "Managed Agent" 원칙 준수**: "credentials never enter the sandbox — fetched by a proxy just-in-time, scrubbed from responses before caller sees them." 구조적 구현:
- docker-compose 의 `boan-sandbox.volumes` 에 `boan-cred-*`, `boan-policy-*` 를 절대 추가하지 않음 (S3 레벨 격리).
- `boan-proxy.resolveTemplateCredentials` 가 `OrgLLMProxyURL` 설정 시 placeholder 를 **그대로 passthrough** (S3 레벨에서 credential 조회 조차 안 함).
- 실제 평문 substitute 는 `boan-org-llm-proxy.credresolver.ResolveAll` 에서만 발생 (S5 의 Cloud Run 메모리, 수 ms).
- `boan-credential-filter` 가 `BOAN_ORG_CREDENTIAL_GATE_URL` 설정 시 `Store.WithGate` 모드로 전환 — register / get / list / revoke 전부 cloud 로 proxy, 로컬 디스크 저장 완전 skip.

**위협 모델**: 로컬 호스트 root 탈취 시에도 credential 평문을 볼 수 없다. 공격자는 최대 S5 의 bearer token 만 획득 가능하며, 그 토큰으로는 gate 의 rate-limit + allowlist + audit 감시 하에서만 요청 가능 (P3 에서 device JWT 로 토큰 탈취 scope 를 더 좁힐 예정).

## 4. 사용자 신뢰 모델 — Deny / Ask / Allow

| 레벨 | G1 적용 | G2/G3 적용 | 모니터링 |
|------|--------|-----------|---------|
| **Allow** | O (무조건) | X (건너뜀) | O |
| **Ask** | O | O (전체) | O |
| **Deny** | O | O (전체) | O |

같은 G3 ask 판정이라도: Allow → 통과, Ask/Deny → 인간 승인 대기.

## 5. G1/G2/G3 가드레일 체계

```
G1 (정규식)     — 전원 무조건 적용. credential, 위험 패턴 정규식 매칭
G2 (헌법+LLM)  — ask/deny만. 보안 LLM이 헌법 기준 판정
G3 (wiki 적응형) — ask/deny만. 과거 결정 학습, 자기진화
                  G3이 주기적으로 G1/G2 자동 개정 제안 (diff → 소유자 수락/거절)
```

## 6. Fail-Closed 원칙

- 네트워크: 화이트리스트 없으면 전부 차단
- 가드레일 LLM 오류: 차단 (heuristic fallback 아님)
- 정책 서버 응답 없음: 차단
- 자격증명 미등록: REDACT
- **OpenClaw 무결성 실패**: sandbox 컨테이너 즉시 종료 (entrypoint 가 sha256/version 검증)

## 7. Supply-Chain 무결성

OpenClaw 는 외부 npm 패키지라 supply-chain 공격(악성 publish, registry 변조)의 표적.
BoanClaw 는 두 단계로 막음:

```
빌드 시점: ARG OPENCLAW_VERSION=2026.4.9 (핀)
           → npm install -g openclaw@${OPENCLAW_VERSION}
           → package.json.version 검사 (mismatch 시 image build FAIL)
           → sha256sum openclaw.mjs > /opt/boanclaw-meta/openclaw.mjs.sha256

런타임:    entrypoint.sh 가 매 시작마다 재계산
           → version mismatch  → fail-closed 종료
           → sha256 mismatch   → fail-closed 종료
           → BOAN_OPENCLAW_ALLOWED_VERSIONS allowlist (있으면)
```

새 OpenClaw 허용 절차: `Dockerfile` 의 `OPENCLAW_VERSION` 갱신 → `scripts/rebuild.sh` → 새 sha256 자동 기록.
