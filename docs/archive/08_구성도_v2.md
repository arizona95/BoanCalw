# BoanClaw 구성도 v2

## 1. 철학: S4 보안 영역 분리와 정보 흐름 제어

### 1-1. S4 구조 — 정보의 영역 분리

모든 데이터와 인프라는 4단계 보안 레벨(S-Level)로 분류된다.

```
S4  │ CSP Zone (Cloud Service Provider)
    │ 조직 정책, 감사 로그, 자산 분류, 서명 키
    │ → 가장 높은 신뢰 영역
────┼──────────────────────────────────────────
S3  │ Host Zone (사용자 PC, 관리 콘솔)
    │ 관리 UI, 프록시, 자격증명 필터, LLM 레지스트리
    │ → 내부 영역, 고객 책임
────┼──────────────────────────────────────────
S2  │ Sandbox Zone (Docker 컨테이너)
    │ OpenClaw 에이전트, 마운트 디렉토리, 코드 실행
    │ → 격리된 작업 환경
────┼──────────────────────────────────────────
S1  │ External Zone (GCP 작업 PC, 외부 API)
    │ 원격 데스크탑(RDP), 외부 LLM API, 인터넷
    │ → 가장 낮은 신뢰 영역
```

### 1-2. 정보 흐름 원칙

```
↓ 낮은 곳으로 흐르면 (S4→S1 방향) = 검사
  - 가드레일 3단계 (헌법 → wiki → 인간 승인)
  - DLP 검사 (패턴 매칭 + LLM 분류)
  - Credential 게이트 (자격증명 마스킹)
  
↑ 높은 곳으로 흐르면 (S1→S4 방향) = 통과
  - 제한 없이 전달
  - 모니터링만 수행
```

### 1-3. 내부/외부 경계

```
┌─────────────────────────────────────┐
│         내부 (고객 책임)             │
│  S4: 정책 서버, 감사                 │
│  S3: 사용자 PC, 관리 콘솔            │
│  S2: Sandbox (Docker)               │
│                                     │
│  Credential 게이트:                  │
│  내부→외부 흐를 때 무조건 적용       │
│  실제 키는 S3에만 존재               │
│  외부에는 {{CREDENTIAL:name}} 플레이스홀더 │
└────────────────┬────────────────────┘
                 │ Network Gate (allowlist)
                 ▼
┌─────────────────────────────────────┐
│         외부                         │
│  S1: GCP 작업 PC, 외부 LLM API      │
│  인터넷                              │
└─────────────────────────────────────┘
```

## 2. 3단계 신뢰 모델: Deny / Ask / Allow

### 2-1. 사용자 신뢰 레벨

소유자(Owner)는 초월적 존재로 제한 없음. 사용자만 아래 3단계 적용.

| 레벨 | 정보의 낮은 흐름 | 가드레일 | 모니터링 |
|------|----------------|---------|---------|
| **Allow** | 허용 | 적용하지 않음 | O (로그만) |
| **Ask** | 가드레일 결과에 따라 | 3-tier 전체 적용 | O |
| **Deny** | 차단 | 같은/높은 레벨만 가능 | O |

### 2-2. 가드레일 3-Tier 판정과 사용자 정책의 교차

같은 가드레일 판정이라도 사용자 정책에 따라 결과가 다르다:

```
가드레일 판정    Allow 사용자    Ask 사용자    Deny 사용자
─────────────  ────────────  ───────────  ───────────
Tier1: allow   → 통과         → 통과        → 통과
Tier1: block   → 차단         → 차단        → 차단
Tier1: ask
 └ Tier2: allow → 통과        → 통과        → 통과
 └ Tier2: block → 차단        → 차단        → 차단
 └ Tier2: ask   → 통과(모니터링) → 인간승인 대기  → 인간승인 대기
```

### 2-3. 가드레일 3-Tier 구조

```
┌─────────────────────────────────────────┐
│  Tier 1: 헌법 가드레일                    │
│  보안 LLM이 (헌법 + 질문)을 한번에 평가    │
│  리턴: (위험도: allow/ask/deny, 답변)     │
│  allow → 즉시 전달                       │
│  deny  → 즉시 차단                       │
│  ask   → ▼                              │
├─────────────────────────────────────────┤
│  Tier 2: Wiki 가드레일 (자기진화)          │
│  과거 결정 50건을 few-shot context로 활용  │
│  HITL training log에서 학습              │
│  allow → 전달  │  deny → 차단            │
│  ask   → 사용자 access_level 확인 → ▼   │
├─────────────────────────────────────────┤
│  Tier 3: 인간(소유자) Approval           │
│  승인/거부 → Tier 2 wiki 학습 데이터      │
│  wiki가 주기적으로 Tier 1 헌법 개정 제안    │
│  (diff → 소유자 수락/반대)                │
└─────────────────────────────────────────┘
```

## 3. 전체 구성도 — 노드와 엣지 그래프

### 3-1. 컴포넌트 노드

```
[S4 Zone]
  (policy-server)     — 정책 배포, 가드레일 LLM, wiki 평가, 헌법 개정
  (audit-agent)       — 감사 로그 수집/저장

[S3 Zone]  
  (admin-console)     — 관리 UI (Gateway Policies, Users, Approvals, Files)
  (boan-proxy)        — MITM 프록시, 3개 게이트 (Input/Credential/Network)
  (credential-filter) — AES 암호화 키 저장소
  (llm-registry)      — LLM 엔드포인트 등록부
  (whitelist-proxy)   — 네트워크 허용목록 프록시

[S2 Zone]
  (boan-sandbox)      — OpenClaw + boan-agent + boan-proxy(내장)
  (mount-dir)         — S3→S2 마운트 디렉토리 (/workspace/boanclaw)

[S1 Zone]
  (gcp-workstation)   — Windows RDP (Guacamole 경유)
  (external-llm)      — Anthropic API, Ollama Cloud 등
  (computer-use)      — Playwright 기반 화면 제어 서비스
```

### 3-2. 정보 흐름 엣지

```
사용자 키보드/클립보드
    │
    ▼
(admin-console) ──[HTTP]──→ (boan-proxy)
    │                           │
    │                    ┌──────┼──────┐
    │                    ▼      ▼      ▼
    │              [Input   [Credential [Network
    │               Gate]    Gate]       Gate]
    │                │         │          │
    │                ▼         ▼          ▼
    │           (guardrail  (credential  (whitelist
    │            client)     filter)      proxy)
    │                │                    │
    │                ▼                    ▼
    │           (policy-server)    (external-llm)
    │            ├─ Tier 1: 헌법 LLM
    │            ├─ Tier 2: Wiki LLM  
    │            └─ Tier 3: Approval 큐
    │
    ▼
(boan-sandbox) ──[proxy내장]──→ (gcp-workstation)
    │                               │
    ├─ OpenClaw 채팅               ├─ RDP 세션
    ├─ Agent 도구 실행              ├─ Computer-Use
    └─ 파일 마운트                  └─ 화면 캡처
```

### 3-3. 게이트별 상세 흐름

```
[Input Gate]
  사용자 입력 ──→ 패턴 검사 (credential regex)
                    │
              credential 감지? ──yes──→ credential_required
                    │no
              S레벨 하향? (src > dst)
                    │yes
              Tier 1: 헌법 LLM ──allow──→ DLP 검사 → 통과
                    │ask
              Tier 2: Wiki LLM ──allow──→ DLP 검사 → 통과
                    │ask
              사용자 정책 확인
                    ├─ allow 사용자 → 통과 (모니터링)
                    └─ ask/deny 사용자 → Tier 3 인간 승인

[Credential Gate]
  프롬프트 ──→ 등록된 키 치환 (실값 → {{CREDENTIAL:name}})
          ──→ 미등록 키 감지 → REDACT + HITL 승인 요청
          ──→ 실행 시점에만 실제 키 주입

[Network Gate]  
  HTTP 요청 ──→ 정책 allowlist 확인
             ──→ 허용 host:port:method만 통과
             ──→ 나머지 전부 차단 (fail-closed)
```

### 3-4. 파일 전송 흐름

```
[File Manager: S2 ↔ S1]

S2 (Sandbox Mount)           S1 (GCP Workstation)
  /workspace/boanclaw/         /workspace/gcp/
       │                            │
       ├──→ S2→S1: 파일 내용         │
       │    Input Gate 검사          │
       │    (credential + 3-tier)    │
       │    통과 시 복사              │
       │                            │
       │    S1→S2: ←──────────────┤
       │    무조건 통과              │
       │    (높은→낮은 방향)         │
       │                            │
       └── 폴더 전송 차단 ──────────┘
```

### 3-5. 인증/SSO 흐름

```
사용자 ──→ (admin-console) /login
              │
              ├─ Email OTP: 도메인 검증 → OTP 발송 → 검증 → 세션
              └─ Google OAuth: hd=도메인 → 콜백 → 도메인 검증 → 세션
              
허용 도메인: org_settings.allowed_domains (동적 변경 가능)
신규 사용자: pending → 소유자 승인 → approved + GCP 작업PC 할당
```

## 4. 보안 게이트 요약

| 게이트 | 위치 | 적용 시점 | 판정 |
|-------|------|---------|------|
| **Input Gate** | boan-proxy | 키보드/클립보드/채팅/computer-use type/파일전송 → S1 | allow / hitl_required / block / credential_required |
| **Credential Gate** | boan-proxy + credential-filter | LLM 호출 시 프롬프트 내 키 마스킹 | 치환 / REDACT / HITL |
| **Network Gate** | boan-proxy (MITM) | 모든 아웃바운드 HTTP | allow (whitelist) / block (default) |
