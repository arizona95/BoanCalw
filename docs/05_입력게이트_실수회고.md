# 05. 입력 게이트 실수 회고

## 범위

이 문서는 `my-gcp` 원격 작업 컴퓨터 화면의 입력 게이트, 포커스 제어, 클립보드 동기화 작업 중 제가 만든 실수와 원인을 기록한다.

범위는 다음으로 한정한다.

- `MyGCP.tsx`의 포커스 제어
- 키보드/클립보드 입력 게이트
- Guacamole iframe과의 상호작용
- 탭 전환 시 세션 유지

## 핵심 결론

가장 큰 실수는 다음 하나로 요약된다.

- `브라우저 포커스`와 `논리적 입력 소유권`을 분리하지 못하고, 둘을 동시에 제어하려고 하면서 포커스 경쟁을 반복해서 만들었다.

즉 원래 목표는:

- 기본 입력은 BoanClaw 입력창이 받는다.
- 원격 Windows는 마우스만 직접 받는다.
- 조합키와 특수키만 원격으로 전달한다.

였는데, 구현 과정에서 실제 브라우저 포커스를 원격 iframe으로 넘기거나, 반대로 마우스 이벤트마다 다시 로컬로 뺏는 식의 상충되는 로직을 여러 번 넣었다.

## 실수 목록

### 1. 포커스 소유권을 하나로 고정하지 못함

제가 한 실수:

- 로컬 입력창
- 숨은 입력 요소
- 부모 `window` keydown
- Guacamole iframe 내부 input sink

이 네 경로가 동시에 입력을 받는 상태를 오래 유지했다.

문제:

- `Enter` 뒤에는 iframe이 포커스를 먹고
- 마우스 이동이나 클릭 뒤에는 다시 로컬이 포커스를 먹고
- 결과적으로 사용자는 포커스가 랜덤하게 오가는 것처럼 보였다.

원인:

- “누가 키보드의 유일한 소유자인가”를 먼저 고정하지 않고, 증상마다 보조 포커스 로직을 덧붙였다.

### 2. `mousemove` / `mouseenter` / `click`를 포커스 복구 트리거로 사용함

제가 한 실수:

- 마우스 포인터가 원격 화면 위를 움직이거나
- 클릭하거나
- 진입할 때

`focusGateInput()`을 호출해 포커스를 강제로 되돌렸다.

문제:

- 원격에서 텍스트를 선택한 뒤 `Ctrl+C`를 하려 하면
- 마우스 움직임 자체가 포커스를 로컬로 다시 뺏어왔다.
- 그래서 복사 동작이 원격이 아니라 로컬 핸들러에 걸렸다.

원인:

- “포커스 복구”와 “마우스 포워딩”을 같은 이벤트 체인에 섞었다.

### 3. 조합키를 보내기 위해 실제 브라우저 포커스를 원격으로 넘기려 함

제가 한 실수:

- `Ctrl/Meta/Alt`가 눌리면 실제 `iframe`, `remoteWindow`, `remoteTarget`에 포커스를 넘기려 했다.

문제:

- `keydown`은 부모에서 잡혀도
- `keyup`은 iframe 안에서 끝나 부모가 못 받는 경우가 생겼다.
- 그 결과 조합키 passthrough 상태가 풀리지 않아 입력창으로 다시 돌아오지 못했다.

원인:

- 조합키 전달은 “원격에 synthetic 이벤트를 보내는 것”이면 충분한데,
- 불필요하게 “실제 브라우저 포커스 이동”까지 시도했다.

### 4. `keyup` 복귀 로직을 신뢰함

제가 한 실수:

- `Control`, `Meta`, `Alt`를 떼는 `keyup`을 부모 창이 반드시 받는다고 가정했다.

문제:

- 실제로는 포커스가 iframe으로 넘어간 순간 부모가 그 이벤트를 못 받았다.
- 그래서 포커스 복귀 조건이 충족되지 않았다.

원인:

- cross-document / iframe 포커스 전이를 너무 낙관적으로 가정했다.

### 5. 포커스 문제를 증상별 `재시도`로 덮으려 함

제가 한 실수:

- `0ms`, `40ms`, `120ms`처럼 여러 번 `focus()` 재시도를 넣었다.

문제:

- 특정 레이스에는 도움이 될 수 있어도,
- 근본 원인인 포커스 경쟁을 해결하지는 못했다.
- 오히려 어느 이벤트가 마지막 포커스를 먹는지 더 추적하기 어려워졌다.

원인:

- 구조를 단순화하기보다, 타이밍 보정으로 해결하려고 했다.

### 6. 입력 게이트와 원격 주입을 분리하지 못함

제가 한 실수:

- 버퍼링, 키 처리, 즉시 키 전달, 조합키 전달, 텍스트 주입, 포커스 복구를 한 컴포넌트 안에서 한꺼번에 계속 수정했다.

문제:

- 하나를 고치면 다른 경로가 깨졌다.
- 예: 복사 문제를 고치다 일반 입력이 새고, 일반 입력을 막다 포커스 복귀가 깨졌다.

원인:

- 상태 머신과 이벤트 경로를 먼저 고정하지 않고 구현부터 계속 흔들었다.

### 7. `aAbBcC` 같은 중복 입력 원인을 늦게 좁힘

제가 한 실수:

- printable 문자 입력 중복 현상을 포커스 문제와 섞어서 봤다.

실제 원인:

- 원격 주입 시 printable 문자에 대해 중복 이벤트를 보내거나,
- Guacamole가 이미 처리하는 경로와 synthetic 이벤트 경로가 겹쳤다.

문제:

- 사용자는 포커스도 안 되고 입력도 두 벌로 들어가는 상태를 겪었다.

원인:

- 입력 경로 중복 여부를 먼저 검증하지 않고, 포커스 문제로만 접근했다.

### 8. 탭 전환 문제를 화면 렌더링 문제와 입력 문제로 동시에 만들었음

제가 한 실수:

- `my-gcp` 탭을 다시 열 때 iframe이 내려갔다 올라가는 문제와
- 입력 포커스 문제를 같은 시기에 같이 건드렸다.

문제:

- 사용자는 “탭 갔다 오면 다시 켜진다”와 “입력이 안 된다”를 동시에 겪었다.
- 원인 분리가 더 어려워졌다.

원인:

- 세션 유지 문제와 입력 소유권 문제를 별도 이슈로 분리하지 않았다.

### 9. 클립보드 동기화를 너무 낙관적으로 봄

제가 한 실수:

- `Ctrl+C`가 전달되면 곧바로 원격 클립보드가 읽힐 것이라고 가정했다.

문제:

- 실제로는
  - 원격 RDP clipboard redirection
  - Guacamole clipboard service
  - 브라우저 clipboard API 권한

세 단계가 모두 맞아야 한다.

원인:

- clipboard는 키 이벤트 하나로 끝나는 문제가 아니라, 별도 데이터 경로라는 점을 충분히 분리하지 못했다.

### 10. 사용자에게 “됐다”고 너무 이르게 말함

제가 한 실수:

- 코드상 원인을 하나 제거했을 때,
- 실제 브라우저에서 체감 증상이 완전히 사라졌는지 확인되기 전에 “됐다”고 말한 경우가 있었다.

문제:

- 같은 문제가 반복 보고되면서 신뢰가 떨어졌다.

원인:

- 브라우저/iframe/Guacamole 조합 문제는 코드 수정 직후에도 실제 런타임 확인이 더 중요하다는 점을 충분히 반영하지 못했다.

## "됐다"고 잘못 말한 사례 10개

아래는 실제로는 문제가 남아 있었는데, 제가 너무 이르게 “고쳤다”, “반영됐다”, “이제 된다”고 말한 사례들이다.

### 1. 마우스 이동 없이도 포커스가 돌아온다고 말한 경우

- 제가 한 말의 취지:
  - `Enter` 후 바로 다시 입력이 될 것이라고 설명함
- 실제:
  - 마우스를 움직여야만 다시 입력이 살아남
- 잘못한 이유:
  - `mousemove`가 사실상 숨은 포커스 복구 트리거였는데, 그 경쟁 구조를 끝까지 못 봤다.

### 2. 클릭만 하면 포커스 문제가 해결됐다고 말한 경우

- 제가 한 말의 취지:
  - 클릭 후 바로 타이핑이 될 것이라고 설명함
- 실제:
  - 클릭이 임시 복구 역할만 했고, 근본 해결이 아니었음
- 잘못한 이유:
  - `click` 경로의 강제 재포커스를 “해결”로 착각했다.

### 3. `Ctrl+C` 복사가 풀렸다고 말한 경우

- 제가 한 말의 취지:
  - 복사는 이제 바로 통과한다고 설명함
- 실제:
  - 마우스 이동/포커스 강탈 때문에 원격 복사가 계속 깨짐
- 잘못한 이유:
  - `Ctrl+C` 키 경로만 보고, 직전 포커스 강탈 경로를 함께 검증하지 않았다.

### 4. 클립보드 동기화가 된다고 말한 경우

- 제가 한 말의 취지:
  - 원격 클립보드가 로컬과 동기화된다고 설명함
- 실제:
  - clipboard redirection, Guacamole event, 브라우저 권한이 동시에 맞지 않으면 실패함
- 잘못한 이유:
  - 데이터 경로 전체가 아니라 일부 이벤트 연결만 보고 완료로 판단했다.

### 5. 일반 문자열이 더 이상 원격으로 새지 않는다고 말한 경우

- 제가 한 말의 취지:
  - 이제 문자는 버퍼로만 간다고 설명함
- 실제:
  - 포커스가 한 번 어긋나면 원격으로 그대로 써지는 경우가 계속 남아 있었음
- 잘못한 이유:
  - “입력 규칙”이 아니라 “현재 포커스 상태”에 결과가 좌우되는 구조를 완전히 제거하지 못했다.

### 6. `aAbBcC` 중복 입력이 해결됐다고 말한 경우

- 제가 한 말의 취지:
  - printable 문자 중복 경로를 줄였으니 끝났다고 설명함
- 실제:
  - 다른 synthetic 이벤트/포커스 경로가 남아 있어 중복이 재발함
- 잘못한 이유:
  - 중복 입력 원인을 하나만 잡고 전체 입력 경로를 다시 검증하지 않았다.

### 7. 다른 탭 갔다 와도 세션이 유지된다고 말한 경우

- 제가 한 말의 취지:
  - iframe을 유지하도록 바꿨으니 재시작되지 않는다고 설명함
- 실제:
  - 탭 복귀 시 재로딩/재연결로 보이는 케이스가 남아 있었음
- 잘못한 이유:
  - 렌더링 유지와 입력/게이트 상태 유지 문제를 분리 검증하지 않았다.

### 8. `Ctrl` 계열일 때만 원격 포커스로 잠깐 넘기면 된다고 말한 경우

- 제가 한 말의 취지:
  - `Ctrl/Meta/Alt`에서만 예외를 두면 정리된다고 설명함
- 실제:
  - 한 번 원격 iframe이 실제 포커스를 먹으면 부모가 `keyup`을 못 받고 복귀가 깨짐
- 잘못한 이유:
  - “실제 브라우저 포커스 이동” 자체가 설계 실수였는데, 그걸 수정안으로 제시했다.

### 9. `keyup` 복귀 로직을 넣었으니 된다고 말한 경우

- 제가 한 말의 취지:
  - modifier keyup에서 다시 돌아오게 했으니 된다고 설명함
- 실제:
  - keyup이 부모가 아니라 iframe 안에서 끝나 복귀가 아예 안 됨
- 잘못한 이유:
  - iframe 환경에서 이벤트가 어디까지 올라오는지 충분히 검증하지 않았다.

### 10. “이제 확인해보면 된다”를 너무 자주 반복한 경우

- 제가 한 말의 취지:
  - 빌드/재배포 후 확인만 하면 된다고 여러 번 설명함
- 실제:
  - 사용자는 같은 증상을 연속으로 다시 확인해야 했음
- 잘못한 이유:
  - 코드 수정 직후의 내부 확신을 실제 사용자 체감 검증보다 앞세웠다.

## 다시 정리한 올바른 상태 모델

입력 상태는 사실상 두 개만 있어야 한다.

- `focus_chat`
- `focus_screen`

하지만 이 둘의 의미는 엄격히 다뤄야 한다.

### focus_chat

기본 상태다.

- 일반 문자 입력
- 버퍼 편집
- 붙여넣기 검사
- 전송 버튼

은 전부 여기서 처리한다.

### focus_screen

지속 상태가 아니라, “원격에 이벤트를 보내는 순간적 동작 모드”여야 한다.

- 마우스 포인터 전달
- 특수키 전달
- 조합키 전달

만 이 모드로 본다.

중요:

- `focus_screen`을 위해 실제 브라우저 포커스를 원격 iframe으로 넘기면 안 된다.
- 즉 “원격 행동 모드”이지, “원격 브라우저 포커스 상태”가 아니다.

## 재발 방지 원칙

### 1. 실제 브라우저 포커스를 원격 iframe으로 넘기지 않는다

원격에는 synthetic event만 보낸다.

### 2. 마우스 이벤트로 포커스를 바꾸지 않는다

- `mousemove`
- `mouseenter`
- `click`
- `pointerup/down`

은 포인터 전달만 하고, 포커스 제어를 하지 않는다.

### 3. 일반 문자는 오직 한 경로로만 들어간다

일반 문자는 무조건 버퍼로만 들어간다.

### 4. 조합키도 포커스 이동 없이 처리한다

- `Ctrl+C`
- `Ctrl+V`
- `Ctrl+A`

는 로컬 포커스를 유지한 채 원격에 이벤트만 전달한다.

### 5. 복귀 조건을 `keyup` 하나에만 의존하지 않는다

특히 iframe이 있는 구조에서는 부모가 `keyup`을 못 받을 수 있다.

### 6. “재시도”는 마지막 수단으로만 쓴다

여러 번 `focus()`를 때리는 건 구조가 맞는지 확인한 뒤에만 보조적으로 쓴다.

### 7. 탭 전환/세션 유지 문제와 입력 문제를 분리한다

- iframe 유지
- 워크스테이션 상태 갱신
- 입력 포커스

이 세 문제는 반드시 따로 본다.

## 앞으로 수정할 때 체크리스트

1. 이번 수정이 실제 브라우저 포커스를 원격으로 넘기는가
2. 마우스 이벤트가 포커스를 바꾸는가
3. 일반 문자 입력 경로가 둘 이상인가
4. 조합키 전달 후 복귀 조건이 iframe `keyup`에 의존하는가
5. 원격 clipboard와 로컬 clipboard를 같은 것으로 착각하고 있는가
6. 코드상 수정과 사용자 체감 수정이 같은지 실제로 다시 확인했는가

## 절대 하지 말아야 할 안티패턴

### 1. `focus()` 재시도를 여러 번 쌓는 방식

예:

- `0ms`
- `40ms`
- `120ms`

처럼 여러 번 `focus()`를 쏘는 로직

왜 안 되는가:

- 구조 문제를 가린다.
- 어떤 이벤트가 마지막 포커스를 먹는지 추적이 더 어려워진다.
- “우연히 되는 케이스”만 늘어난다.

원칙:

- `focus()` 재시도는 구조가 이미 맞는 경우에만 보조 수단으로 쓴다.
- 문제의 1차 해결책으로 쓰지 않는다.

### 2. 마우스 이벤트를 포커스 복구 트리거로 쓰는 방식

예:

- `mousemove`
- `mouseenter`
- `click`
- `pointerup/down`

뒤에 `focusGateInput()`을 붙이는 것

왜 안 되는가:

- 선택, 드래그, 복사 같은 정상 원격 작업을 깨뜨린다.
- “움직이면 된다”, “클릭하면 된다” 같은 랜덤 체감을 만든다.

원칙:

- 마우스 이벤트는 포인터 전달만 한다.
- 포커스 복구와 섞지 않는다.

### 3. 조합키 전달을 위해 실제 브라우저 포커스를 iframe으로 넘기는 방식

예:

- `iframe.focus()`
- `remoteWindow.focus()`
- `remoteTarget.focus()`

왜 안 되는가:

- 부모가 `keyup`을 잃는다.
- iframe 내부 hidden input sink와 경쟁이 시작된다.
- 복귀가 마우스/클릭 같은 우연한 이벤트에 의존하게 된다.

원칙:

- 조합키는 원격에 synthetic event만 보낸다.
- 실제 브라우저 포커스는 로컬 입력에 둔다.

### 4. `keyup` 하나만 믿고 복귀를 설계하는 방식

예:

- `Ctrl`을 떼는 순간 부모가 반드시 `keyup`을 받을 것이라고 가정

왜 안 되는가:

- iframe 안으로 이벤트가 들어가면 부모가 못 받을 수 있다.
- 핵심 상태 해제가 영원히 안 되는 상태가 된다.

원칙:

- 부모가 항상 관찰 가능한 경로만 상태 해제에 사용한다.

### 5. fallback을 계속 누적하는 방식

예:

- 숨은 textarea
- 부모 `window keydown`
- iframe 포커스 보정
- synthetic event fallback
- clipboard polling fallback

을 계속 덧붙이는 것

왜 안 되는가:

- 입력 경로가 여러 개가 되어 서로 충돌한다.
- 한 버그를 고치면 다른 경로에서 재발한다.

원칙:

- fallback을 넣기 전에 기존 경로를 제거한다.
- 입력 경로는 가능한 한 하나로 줄인다.

### 6. 임시 해결을 영구 로직으로 굳히는 방식

예:

- 디버깅용으로 넣은 강제 `focus()`
- 특정 케이스만 막는 `preventDefault`
- “일단 되게 하자” 분기

왜 안 되는가:

- 코드가 증상 대응 모음집이 된다.
- 누구도 실제 입력 경로를 설명하지 못하게 된다.

원칙:

- 임시 코드는 제거 조건이 없는 한 넣지 않는다.

### 7. 상태 머신 없이 이벤트 핸들러만 고치는 방식

예:

- 문제 생길 때마다 `keydown`, `keyup`, `click`, `pointermove`, `paste` 하나씩 덧대는 것

왜 안 되는가:

- 이벤트 간 상태 전이가 사라진다.
- 코드와 실제 체감이 분리된다.

원칙:

- 먼저 `focus_chat`, `focus_screen` 상태를 적는다.
- 그 다음 각 이벤트가 어느 상태에서 무엇을 바꾸는지 적는다.
- 마지막에 코드를 고친다.

### 8. 코드상 원인 하나 제거하고 바로 “됐다”고 말하는 방식

왜 안 되는가:

- 브라우저, iframe, Guacamole, clipboard는 단일 수정으로 끝나지 않는다.
- 사용자는 같은 증상을 반복 확인하게 된다.

원칙:

- 최소한 아래 시나리오를 통과해야 “됐다”고 말한다.
- `일반 문자 -> Enter -> 다시 일반 문자`
- `선택 -> Ctrl+C`
- `Ctrl+V`
- `탭 전환 후 복귀`

### 9. “복잡한 fallback이 있으니 언젠가 맞겠지”라고 생각하는 방식

왜 안 되는가:

- 복잡도는 문제를 해결하지 않는다.
- 설명할 수 없는 로직이 늘어난다.

원칙:

- 실패하면 fallback을 늘리지 말고 경로를 줄인다.

### 10. 증상별 보정으로 구조 문제를 덮는 방식

예:

- “마우스 움직이면 돌아오게”
- “클릭하면 다시 잡게”
- “3번 재시도”
- “keyup에서 풀리게”

를 계속 덧붙이는 것

왜 안 되는가:

- 입력 소유권이 누구에게 있는지 끝까지 명확해지지 않는다.

원칙:

- 구조 문제는 구조로 해결한다.
- 증상 보정은 마지막에만 넣는다.

## 결론

이번 입력 게이트 작업에서 가장 큰 실패는 “증상별 보정”을 계속 덧붙이면서, 포커스와 입력 소유권의 단일 원칙을 끝까지 지키지 못한 것이다.

앞으로는 다음 한 문장만 기준으로 수정해야 한다.

- `브라우저 포커스는 로컬 입력이 계속 잡고, 원격에는 필요한 이벤트만 보낸다.`
