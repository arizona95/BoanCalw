## 2. 기본 보안요건 (BR)

### BR-01 조직 정책 배포 서버

**요건**

- 기업 단위로 에이전트 설정/정책을 중앙 배포하는 서버가 있어야 함
- 엔드포인트 예: `https://policy.boanclaw.corp/org/{org_id}/policy.json`
- 에이전트 기동 시 정책 서버에 체크인 → 정책 fetch → 적용
- 정책 서버 미응답 시 에이전트 기동 차단 (fail-closed)
- 정책 서명 검증 (서버 private key로 서명, 에이전트가 public key로 검증)

**배포 정책 항목**

- 허용 모델 목록 (model allowlist)
- 허용 BoanClaw 버전 범위 (version allowlist)
- 허용 네트워크 도메인 (network policy)
- 채널 활성화 여부 (Slack/Teams/WhatsApp 등)
- 감사로그 수집 endpoint
- 기능 플래그 (feature flags)
- 라이선스 정보 및 만료일

**참조:** Antigravity의 antigravity-unleash.goog (조직설정 엔드포인트)

### BR-02 프로세스 식별 및 DRM

**요건**

- 기업 환경에서 인가된 BoanClaw 인스턴스 외 다른 claw 툴 실행 차단 가능해야 함

**구현 방식**

- a) 바이너리 서명: 코드 서명(code signing)으로 변조 감지
- b) 프로세스명 해시: 실행 시 고유 org_token 기반 프로세스 이름 생성 (예: `boanclaw-a3f9b2` — 조직마다 다른 해시)
- c) 엔드포인트 등록: 기동 시 policy 서버에 instance_id 등록
- d) OS 정책 연동: Windows(AppLocker/WDAC), Linux(SELinux/AppArmor) — 허가된 프로세스 해시만 실행 허용

- 조직 IT 관리자가 MDM/GPO를 통해 다른 claw 바이너리 실행 차단 가능한 문서/스크립트 제공

**참조:** NemoClaw의 openshell-sandbox 프로세스 격리 방식

### BR-03 OneCLI 자격증명 프록시

**요건**

- AI 에이전트는 실제 API 키를 절대 알 수 없어야 함
- 에이전트가 보내는 모든 AI API 요청은 OneCLI 프록시를 경유
- OneCLI가 요청 헤더에 실제 자격증명을 주입하여 upstream으로 전달
- **배치 모델 (NemoClaw 패턴):** OneCLI는 **별도 컨테이너가 아니라** `boan-sandbox` Pod **내부에서 동작하는 내부 프로세스**로 실행한다. 에이전트 컨테이너/프로세스에는 자격증명이 없으며, 로컬 프록시에만 연결한다.
- 에이전트는 `ANTHROPIC_BASE_URL=http://localhost:8083` (또는 동일 Pod 내 OneCLI 리스닝 포트)만 설정하면 된다. 실제 키는 Pod 내부의 OneCLI 프로세스가 메모리·Vault 연동으로만 취급한다.

**지원 자격증명 유형**

- API Key (`x-api-key` 헤더 주입)
- OAuth Bearer Token
- AWS Signature V4 (SageMaker, Bedrock)
- Azure AD Token (Azure OpenAI)
- GCP Service Account Token

**추가 기능**

- 요청/응답 내용 감사 로그 기록 (BR-09와 연동)
- 모델 이름 rewrite (에이전트가 요청한 모델 → 실제 허용 모델로 교체)
- Rate limiting (조직별, 사용자별 호출 제한)

**참조:** NanoClaw credential-proxy.ts + NemoClaw openshell-managed 방식 통합

### BR-04 CSP Vault 연동

**요건**

- OneCLI가 사용하는 API 키는 고객 책임 CSP 시크릿 저장소에서 조달
- k8s Secret → CSP Vault 연동 (External Secrets Operator 또는 CSI Driver)

**지원 CSP**

- a) Azure Key Vault (주)
- b) AWS Secrets Manager
- c) GCP Secret Manager
- d) HashiCorp Vault (On-prem)

- 시크릿 자동 갱신 (TTL 기반 rotation)
- 시크릿 접근 감사로그 CSP 측에 자동 기록

**k8s 구현**

- SecretProviderClass (CSI Driver)로 Pod에 시크릿 마운트
- ServiceAccount + IRSA/Workload Identity로 최소 권한 접근

**참조:** [BR-03] OneCLI 연동, External Secrets Operator 패턴

### BR-05 네트워크 정책 게이트

**요건**

- 에이전트 컨테이너의 모든 아웃바운드는 Network Policy Gate를 반드시 경유
- 허용 목록에 없는 도메인/IP로의 연결 차단 (fail-closed)
- TLS Termination: Gate가 TLS를 종료하여 내용 검사 가능
- 정책은 조직 정책 서버(BR-01)에서 배포

**정책 항목**

- 허용 도메인 (host: api.openai.com, port: 443)
- 허용 메서드/경로 (GET/POST /v1/*)
- 사내 provider 도메인
- Slack/Teams webhook 도메인

**구현**

- Envoy 기반 사이드카 프록시 (Envoy xDS로 동적 정책 갱신)
- 또는 OpenShell 방식의 k8s NetworkPolicy + 전용 프록시 Pod
- Linux: iptables/nftables로 프록시 우회 차단

**참조:** NemoClaw OpenShell 네트워크 정책(openclaw-sandbox.yaml) 방식 개선

### BR-06 발신자 허용 목록

**요건**

- 허가된 사용자/채널에서만 에이전트에게 원격 요청 가능
- 채널별 (Slack/Teams/WhatsApp/CLI) 허용 사용자 목록 관리
- 조직 정책 서버(BR-01)에서 중앙 배포
- 동적 갱신 (에이전트 재시작 없이 allowlist 업데이트)

**차단 방식**

- drop 모드: 차단된 발신자 메시지는 DB에도 저장 안 함 (NanoClaw 방식)
- silent 모드: 저장하되 응답만 안 함
- notify 모드: 차단 사실을 관리자에게 알림

**참조:** NanoClaw sender-allowlist.ts + OpenClaw groupPolicy

### BR-07 원격 권한 RBAC

**요건**

- 채널/사용자별로 에이전트가 행사할 수 있는 권한 등급 분리

**권한 등급**

- LEVEL 0 - read_only: 질의응답, 코드 읽기만 가능. 파일 쓰기/명령 실행 불가
- LEVEL 1 - code_edit: 코드 수정 가능. 시스템 명령 실행 불가
- LEVEL 2 - local_exec: 허용된 명령어 목록 실행 가능 (allowlisted commands)
- LEVEL 3 - full: 제한 없음 (main 채널, 관리자만)

**구현**

- 조직 정책에 user_id → permission_level 매핑
- 에이전트 기동 시 RBAC 컨텍스트 주입
- 레벨 초과 작업 요청 시 거부 + 감사로그 기록

**참조:** NanoClaw remote-control (main 그룹만 허용) 방식 확장

### BR-08 사내/사외 Provider 이중 라우팅

**요건**

- 에이전트 요청이 사외 AI API로 나가기 전 사내 Provider를 한 번 경유

**사내 Provider 역할**

- a) 컨텐츠 필터링 (민감정보 마스킹, DLP)
- b) 요청 승인/거부 (정책 위반 프롬프트 차단)
- c) 캐싱 (동일 요청 재사용)
- d) 모델 라우팅 (비용/성능에 따라 모델 선택)

**흐름**

```
에이전트
  → OneCLI Proxy (BR-03)
  → 사내 Provider (DLP + 필터 + 라우팅)
    → 사내 LLM (NIM/vLLM) [보안 데이터용]
    → 사외 LLM (OpenAI/NVIDIA/Anthropic) [일반 데이터용]
```

**사내 Provider 인터페이스**

- OpenAI-compatible API (drop-in replacement)
- 요청/응답 전처리 훅 (middleware chain)

**참조:** NemoClaw blueprint.yaml profiles (default/ncp/nim-local/vllm) 방식 확장

### BR-09 감사로그 수집

**요건**

- 에이전트의 모든 활동을 감사 로그로 기록
- 로그는 고객 책임 CSP(S3/Azure Blob/GCS)에 저장
- 고객이 직접 보유 → 벤더 종속 없음

**로그 항목**

- 질의(prompt) + 응답(response) 내용 (선택적 암호화)
- 도구 호출 (tool_call): 파일 읽기/쓰기/실행 명령
- 네트워크 요청 (외부 연결 시도)
- 권한 초과 시도 (BR-07 거부 기록)
- 정책 위반 차단 기록 (BR-05, BR-06)
- 세션 시작/종료

**수집 구조**

```
OneCLI Proxy → OpenTelemetry SDK → Prometheus
  → Prometheus Remote Write → Grafana Mimir / Thanos
  → CSP Object Storage (S3/Azure Blob)
```

**로그 포맷**

- JSON Lines (OTEL LogRecord 스펙)

```json
{
  "timestamp": "2026-03-19T00:00:00Z",
  "org_id": "sds-corp",
  "user_id": "user@sds.com",
  "session_id": "abc123",
  "event_type": "tool_call | llm_request | policy_block | ...",
  "payload": { },
  "hash": "sha256(payload)"
}
```

**참조:** NanoClaw의 로그 부재 → 완전 신규 설계 필요

### BR-10 버전 관리

**요건**

- 조직 정책 서버(BR-01)에서 허용 버전 범위 배포
- 허용 범위 밖 BoanClaw 버전은 기동 차단
- 에이전트 기동 시 버전 체크인 → 정책 서버 응답으로 허용/차단 결정

**정책 예시**

```json
{
  "version_policy": {
    "min_version": "1.2.0",
    "max_version": "1.x.x",
    "blocked_versions": ["1.3.1", "1.3.2"],
    "required_by": "2026-06-01"
  }
}
```

**업데이트 채널**

- stable: 기본 (기업 검증 완료)
- beta: 선택적
- security: 긴급 패치 (관리자 승인 없이 강제 적용 가능)

**참조:** Antigravity antigravity-auto-updater 방식 참고

### BR-11 파일시스템 격리

**요건**

- Linux: Landlock (커널 LSM) — NemoClaw 방식
- Windows(WSL): Docker + 볼륨 통제 — NanoClaw 방식

**공통**

- 에이전트 작업 디렉토리만 R/W 허용
- `.ssh`, `.aws`, `.env` 등 시크릿 경로 차단
- project 코드는 READ-ONLY (코드 수정 후 재시작 탈출 방지)
- mount-allowlist는 에이전트 접근 불가 위치에 저장

**참조:** NemoClaw Landlock + NanoClaw mount-security.ts 통합

## 3. Zone 보안요건 (ZR)

### ZR-01 S-Level 등급 분류 체계

**요건**

- 인프라와 데이터를 단일 S-Level 체계로 통합 관리
- 숫자 높을수록 보안 통제 수준 높음 (S4 최고 · S1 최저)
- 데이터 Sn은 인프라 Sn 이상에서만 존재·이동 가능
- 등급이 낮아지는 방향(외부로): 게이트 통과 필수
- 등급이 높아지는 방향(내부로): 자유

**S-Level 정의**

- S4 = 조직 CSP (기밀 · 절대 외부 유출 불가)
- S3 = 내 호스트 PC (민감 · 조직 내부까지만)
- S2 = 내 호스트 안 Docker (일반 업무 · 비식별화 후 외부 가능)
- S1 = 개인 클라우드 (공개 · 자유 이동)

**S-Level 분류 기준**

- `boan-asset-constitution` (S4 보관)에서 관리
- 분류 대상: Image / Text / File (계층형)
- AI 관리 사이클: 헌법 입력 → 애매 항목 문제 제시 → 사용자 정답 입력 → 헌법 수정
- 각 호스트는 분류 기준 원본 미보유 (S4에서 배포 받음)

### ZR-02 3-Gate 보안 경계

**요건**

- S-Level 경계마다 성격이 다른 게이트를 배치
- 게이트별 신뢰 주체와 책임 귀속이 명확해야 함

**S4↔S3 게이트 (시스템적 제어)**

- 신뢰 주체: 시스템 (수학적 보장)
- KMS/HSM 암호화·복호화 필수
- ed25519 정책 서명 검증 (실패 시 미적용 + 알림)
- Credential TTL 자동 순환 (ok/expired/missing/invalid_expires/unresolved_ref)
- 정책 서버 미응답 시 fail-closed

**S3↔S2 게이트 (인간 검수)**

- 신뢰 주체: 개발자 본인
- `boanclaw mount` 명령 = 개발자가 의식적으로 S2 진입 허용
- S4 데이터 포함 감지 시 경고 표시 → 강제 진행 시 감사 로그
- 책임: 인간이 선택한 것 = 인간이 책임

**S2↔S1 게이트 (조직 검토 + 필터링)**

- 신뢰 주체: 조직 (정책 서버)
- boan-proxy가 자동 집행 (개인 판단 개입 없음)
- S4·S3 데이터 → 차단
- S2 데이터 → PII 마스킹 후 통과
- S1 데이터 → 자유 통과

### ZR-03 듀얼 에이전트 라우팅

**요건**

- boan-proxy가 DLP 판정 결과로 에이전트 자동 선택
- 보안용 비서 (S2 Docker): 민감 작업 전담
- 실용성 비서 (S1 클라우드): 일반 작업 전담
- 사용자는 `--secure` / `--cloud` 플래그로 수동 강제 가능

**보안용 비서 (S2)**

- 대상: S4·S3 데이터 포함 요청 · 민감 코드 · 내부 문서
- 구현: OpenClaw 로컬 · k3s sandbox 내부 실행
- 아웃바운드: boan-proxy 경유만 허용 (직접 외부 호출 금지)

**실용성 비서 (S1)**

- 대상: S1·S2 데이터만 포함 요청 · 일반 코딩 · 리서치
- 구현: SkyClaw / OpenClaw Cloud / 미래 AI 서비스
- 내부 통제: S1 진입 후 없음 (게이트에서 이미 판정 완료)

### ZR-04 S-Level별 LLM 거버넌스

**요건**

- 데이터가 머무는 S-Level에 따라 사용 가능한 LLM이 달라야 함
- `boan-llm-registry`가 LLM 객체 목록과 scope 접근 제어 관리

**S3 호스트 직접**

- 사내 Security LLM 전용
- 내 컴 전체 소스 접근 허용 (Claude Code 스타일)
- 외부 LLM 호출 절대 불가

**S2 Docker**

- 사내 Security LLM 기본
- 외부 LLM 필요 시: Security LLM → DLP 비식별화 → External LLM (커넥터)
- 외부 LLM은 Security LLM의 도구로 등록된 형태 (직접 호출 불가)
- 커넥터 등록: curl 명령 → boan-llm-registry → 테스트 → Security LLM 바인딩
- 같은 curl 명령이더라도 이름 다르면 별도 객체로 등록 가능

**S1 클라우드**

- External LLM 자유 사용
- 추가 모니터링 없음 (게이트 통과 = S1 판정 완료)

### ZR-05 boan-proxy 경량 단일 컨테이너

**요건**

- 호스트에 상시 실행되는 단일 Docker 컨테이너
- 메모리 512MB 이하 · CPU 0.5core 상시
- 서비스 무관 동작: 어떤 AI 서비스든 HTTP/HTTPS 인터셉트
- 새 서비스 추가 시 코드 변경 없음 (Org Policy Whitelist 등록만)
- fail-closed: proxy 다운 시 외부 트래픽 전체 차단

**내부 구성**

1. Identity / RBAC (Tool 단위 auth_policy)
2. TLS 인터셉터 (MITM · 자체 CA · 재암호화)
3. Network Policy Gate (Egress Whitelist · fail-closed)
4. DLP 이중 탐지 엔진 (Rule ~0ms + Local LLM ~1-2s)
5. 라우터 (보안용 비서 ↔ 실용성 비서)
6. Credential 주입 (메모리 전용 · 디스크 저장 금지)
7. Rate Limit + Audit (OTEL → S4 Logging)

**트래픽 라우팅 방식**

- Option A: HTTP_PROXY / HTTPS_PROXY 환경변수
- Option B: iptables transparent proxy (에이전트 수정 불필요)

## 4. 회복탄력성 요건 (RR)

### RR-01 Git 강제 커밋 + 파괴적 명령 차단

**요건**

- 에이전트가 파일 수정 시 반드시 git commit 후 작업 허용
- 세션 종료 시 미커밋 변경사항 자동 커밋 (auto-commit)
- 파괴적 git 명령어 차단 (에이전트 레벨 + 컨테이너 레벨 이중 차단)

**차단 명령어 목록**

- `git reset --hard` (작업 내용 완전 소멸)
- `git push --force` (리모트 히스토리 덮어쓰기)
- `git push --force-with-lease`
- `git rebase -i` (히스토리 변조)
- `git commit --amend` (마지막 커밋 변조)
- `git clean -fd` (추적되지 않은 파일 삭제)
- `git checkout -- .` (워킹트리 전체 되돌리기)
- `rm -rf .git` (레포지토리 파괴)

**구현 방식**

- a) git wrapper 스크립트 (컨테이너 내부 PATH에 먼저 등록) — `/usr/local/bin/git` → 차단 명령 감지 → 거부 + 감사로그
- b) Landlock/볼륨 통제로 `.git/objects` 쓰기 제한 (에이전트는 git add/commit만 허용, rebase/reset은 OS 레벨 차단)
- c) 원격 브랜치 보호: main/master → Protected branch (force-push 불가), 에이전트 전용 브랜치 패턴: `boanclaw/{session_id}`

**자동 커밋 흐름**

- 에이전트 작업 시작 → `git checkout -b boanclaw/{session_id}`
- 파일 수정마다 → `git add` + `git commit -m "[boanclaw] auto: {변경요약}"`
- 세션 종료 → 미커밋 변경 자동 커밋 → PR 생성 또는 브랜치 보존

### RR-02 컨테이너 오케스트레이션 + 회복

**요건**

- 에이전트 컨테이너를 오케스트레이터(Orchestrator)가 관리
- 비정상 종료 시 자동 재시작 (단, 보안 위반으로 강제 종료된 경우는 재시작 차단)
- 오케스트레이터가 에이전트 컨테이너를 강제 종료 가능 (정책 위반, 리소스 초과 등)

**컨테이너 구성 (BR-03 정합)**

- Orchestrator Container: 에이전트 Pod 생명주기, 헬스체크, 정책 위반 kill
- **boan-sandbox Pod:** 에이전트 + **OneCLI(동일 Pod 내 내부 프로세스)** — 별도 OneCLI 컨테이너 없음
- Audit Container (또는 사이드카): 로그 수집/전송

**재시작 정책**

- 일반 크래시: 자동 재시작 (최대 3회, 지수 백오프)
- OOM: 재시작 (메모리 제한 조정 후)
- 보안 위반 종료: 재시작 차단 + 관리자 알림
- 정책 변경: 오케스트레이터가 롤링 재시작

**리소스 제한 (기본값, 정책으로 조정 가능)**

- CPU: 2 core 제한
- Memory: 4GB 제한
- PID: 512개 제한 (fork bomb 방지)
- Network: OneCLI 프록시 로컬 포트(예: 8083) 및 정책 허용 경로만

### RR-03 에이전트 Watchdog

**요건**

- Watchdog 프로세스가 에이전트 Pod(에이전트 + Pod 내 OneCLI) 및 Audit 등 관련 컴포넌트를 독립 감시
- 각 컴포넌트의 헬스체크 실패 시 재시작/알림
- Watchdog 자체가 죽으면 시스템 전체 종료 (단일 장애점 방지)

**헬스체크 항목**

- OneCLI Proxy: HTTP GET `/healthz` (10초마다) — Pod 내부 또는 localhost
- Network Gate: Envoy admin `/ready` (10초마다)
- Audit Agent: OTEL export 성공 여부 (60초마다)
- Policy Client: 정책 TTL 만료 감시 (30분마다 갱신)

**장애 등급**

- WARN: 헬스체크 1회 실패 → 재시도
- ERROR: 3회 연속 실패 → 컨테이너 재시작 + 알림
- FATAL: OneCLI/Gate 복구 불가 → 에이전트 전체 중단 + 관리자 알림

### RR-04 작업 체크포인트 및 재개

**요건**

- 에이전트가 장시간 작업 중 비정상 종료되어도 작업 재개 가능
- 세션 상태(대화 히스토리, 진행 중인 도구 호출)를 주기적으로 체크포인트 저장

**저장 항목**

- 세션 ID + 대화 히스토리 (SQLite → 영속 볼륨)
- 진행 중인 파일 작업 목록 (어느 파일을 수정 중이었는지)
- 마지막 성공 git commit hash

**재개 흐름**

- 에이전트 재시작 → 마지막 체크포인트 로드 → git status 확인 (미커밋 변경 존재 시 자동 커밋) → 세션 히스토리 복원 → 사용자 확인

### RR-05 설정 스냅샷 롤백

**요건**

## 5. 보안 갭 분석 (외부 참조)

### [참조-01] EdgeClaw: DLP 이중 탐지 엔진

EdgeClaw는 모든 요청을 S1/S2/S3 3단계로 실시간 분류 후 라우팅한다.

| 등급 | 의미 | 처리 전략 | 예시 |
|------|------|-----------|------|
| S1 | 안전 | 외부 클라우드 LLM 직통 전달 | "봄에 관한 시 써줘" |
| S2 | 민감 | 비식별화(Desensitize) 후 전달 | 주소, 전화, 이메일 |
| S3 | 기밀/사적 | 로컬 LLM 전용 처리 (외부 불가) | 급여명세, PW, SSH키 |

**이중 탐지 엔진**

1. Rule Detector — 키워드 + Regex, ~0ms — API키, DB커넥션, PEM헤더
2. Local LLM Detector — 시맨틱 이해, ~1–2s — 문맥적 민감 데이터

**S2 비식별화 흐름:** 원문 → Local LLM 탐지 → PII JSON 추출 → `[REDACTED:TYPE]` 치환 → Privacy Proxy → 클라우드 LLM → 응답 역치환 반환

**BoanClaw 현재 상태**

- boan-policy-proxy ④ DLP: "민감 정보 탐지/마스킹 · Critical Asset 룰"
- boan-whitelist-proxy: "Pass/Block" 이진 결정
- 탐지 엔진 구현 방식이 미정의
- S2 비식별화 후 외부 LLM 전달 경로 없음 (현재는 차단 or 통과 뿐)
- 이중 탐지(Rule + LLM) 구조 없음

**갭 (GAP)**

- **GAP-01** DLP 탐지 엔진 구현 미정의 (Rule vs LLM 이원화 미반영)
- **GAP-02** S2 비식별화 후 클라우드 전달 경로 없음 — 현재는 "민감 → 차단"이라 사내 LLM 없으면 기능 불가
- **GAP-03** S3 기밀 데이터 전용 로컬 처리 경로 없음 — 외부 LLM으로 나가선 안 되는 데이터 기준 미정의

**반영 권고**

1. boan-policy-proxy ④ DLP를 Rule Detector(~0ms) + Local LLM Detector(~1-2s) 구조로 구체화하고, 두 엔진 AND/OR 조합 가능(checkpoints 설정).
2. 라우팅 결정을 3단계로 분리: S1 → 사용성 트랙(boan-llm-registry); S2 → PII 마스킹 후 사용성 트랙(원문 비저장, 역치환 응답); S3 → 보안 트랙(boan-whitelist-proxy) → 사내 LLM 전용(외부 차단 강제).
3. boan-asset-constitution에 S1/S2/S3 경계 정의 포함.

**우선순위:** P0 (핵심 DLP 라우팅 설계의 빈 구간)

### [참조-02] openclaw-multitenant: No-Secrets-On-Agent

**OCMT 기법**

- **Rule 1: No Secrets on Agent Server** — Management Server(신뢰)에 API 키·사용자 DB·Secrets Vault; Agent Server(비신뢰)는 디스크에 키 없음, 키는 컨테이너 메모리에만, 컨테이너끼리 서로 못 봄.
- **Rule 2: Container Isolation** — 격리 네트워크, 격리 파일시스템, 리소스 한도, No Docker socket.
- **Rule 3: Container Escape Blast Radius** — 탈출 시 해당 사용자 1명 토큰만, 다른 사용자·Vault·Management DB 접근 불가.

**BoanClaw 현재 상태**

- boan-credential-filter: Org Policy Server에서 키 주입
- boan-sandbox Pod에 BOAN_TLS_CERT/KEY 환경변수 주입
- 주입된 API Key가 파일로 저장되는지 메모리만인지 미정의
- Docker socket 마운트 금지 명시 없음
- 컨테이너 탈출 시 피해 범위 원칙 없음

**갭 (GAP)**

- **GAP-04** 주입 Credential 저장 방식 미정의 (디스크 vs 메모리)
- **GAP-05** Docker socket 마운트 금지 원칙 미명시
- **GAP-06** 컨테이너 탈출 시 피해 범위(blast radius) 원칙 미정의

**반영 권고**

1. boan-credential-filter 주입 원칙 강화: API Key는 환경변수만(파일 저장 금지), `/proc/{pid}/environ` 노출 방지(non-root + Landlock `/proc` 제한), 세션 종료 시 unset.
2. Sandbox 격리 규칙: Docker socket 절대 금지, k3s NetworkPolicy default-deny, PID namespace 격리.
3. 탈출 시 획득 가능 범위·불가능 범위를 아키텍처에 명문화.

**우선순위:** P1

### [참조-04] kclaw: Credential TTL + Tool 인증 정책

**kclaw 기법**

1. **Credential 상태 코드:** `ok`, `missing_credential`, `invalid_expires`, `expired`, `unresolved_ref` — expires 부재 시 유효 간주, tokenRef도 expires 우회 불가, 만료 프로필 즉시 사용 불가.

**반영 권고**

1. boan-credential-filter에 상태 코드 도입 및 상태별 처리(missing/invalid_expires → 기동 차단, expired → 갱신 시도 등).

**우선순위:** Credential 상태코드 = P1 · Tool 인증정책 = P1

## 7. P0 반영 요약 — DLP 재설계

**현재 (v3)**

- ④ DLP + LLM 라우팅 결정: 민감 정보 탐지/마스킹 · Critical Asset 룰 → [보안 트랙] boan-whitelist-proxy → [사용성 트랙] boan-llm-registry

**개선안 (v4 후보)**

- ④ DLP + 3단계 분류 + LLM 라우팅

**탐지 단계**

- ① Rule Detector (정규식 · ~0ms): API Key / DB URL / PEM / 전화번호 / 주민번호 패턴
- ② Local LLM Detector (소형모델 · ~1-2s)
- 두 엔진 결과 결합 → S1 / S2 / S3 판정

**라우팅**

- **S1 (안전)** → [사용성 트랙] boan-llm-registry → 사내/외부 LLM
- **S2 (민감)** → PII 마스킹 → [사용성 트랙] → 응답 역마스킹
- **S3 (기밀)** → [보안 트랙] boan-whitelist-proxy → 사내 LLM 전용 (외부 LLM 전달 iptables 차단)

**Critical Asset (boan-asset-constitution 기준):** S3 강제 분류 + 추가 마스킹 + OTEL 알림
